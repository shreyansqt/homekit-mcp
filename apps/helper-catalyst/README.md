# HomeKit MCP Helper App

Minimal Mac Catalyst proof-of-life skeleton for a native-feeling helper that requests Apple Home access through HomeKit.

## Current status

The app builds and tests successfully as Mac Catalyst with code signing disabled. It also starts a localhost-only read-only inventory server at `http://127.0.0.1:8765`.

Endpoints:

- `GET /health`
- `GET /`
- `GET /inventory`
- `POST /mcp` — current simple MCP-style bridge stub. Read-only inventory returns JSON; mutation skeletons default to `plan` and require `mode: "apply"` plus `confirm_apply: true` before calling HomeKit mutation APIs.

```bash
xcodebuild \
  -project HomeKitMCPHelper.xcodeproj \
  -scheme HomeKitMCPHelper \
  -destination 'platform=macOS,variant=Mac Catalyst' \
  CODE_SIGNING_ALLOWED=NO \
  build
```

A normal signed build requires your own Apple developer team. The checked-in project intentionally does **not** contain a development team ID or signing identity. Provide one locally, for example:

```bash
xcodebuild \
  -project HomeKitMCPHelper.xcodeproj \
  -scheme HomeKitMCPHelper \
  -destination 'platform=macOS,variant=Mac Catalyst' \
  DEVELOPMENT_TEAM=YOURTEAMID \
  build
```

## Important UI note

The helper now owns HomeKit and the localhost server from app-level state, with automatic/sudden termination disabled. Closing the visible SwiftUI status window should no longer be treated as quitting the helper; the localhost HomeKit bridge remains alive until the process is explicitly quit or stopped by launchd.

This is still not a true status-item menu extra. SwiftUI `MenuBarExtra` and AppKit `NSStatusItem`/`NSMenu` are unavailable to this Mac Catalyst target, while HomeKit access is not available to a normal native AppKit command-line helper. We intentionally do **not** hide the Catalyst app with `LSUIElement`, because HomeKit privacy prompts need normal app presentation.

A tiny native AppKit wrapper has been added in [`../menubar`](../menubar). It owns the actual menu-bar status item and talks to this Catalyst helper over localhost for health/inventory; HomeKit access remains here in the Catalyst app.

## Files

- `HomeKitMCPHelper.xcodeproj` — generated Xcode project.
- `project.yml` — XcodeGen spec used to generate the project.
- `HomeKitMCPHelper/HomeKitMCPHelperApp.swift` — Catalyst app entry.
- `HomeKitMCPHelper/MenuBarView.swift` — compact status/actions view for the optional visible window.
- `HomeKitMCPHelper/StatusBarController.swift` — Catalyst seam documenting the status-item limitation and preserving a place for a future AppKit wrapper integration.
- `HomeKitMCPHelper/HomeStore.swift` — `HMHomeManager` proof-of-life store.
- `HomeKitMCPHelper/Info.plist` — generated by XcodeGen; includes `NSHomeKitUsageDescription`.
- `HomeKitMCPHelper/HomeKitMCPHelper.entitlements` — includes `com.apple.developer.homekit`.
- `HomeKitMCPHelperTests/InventorySummaryTests.swift` — unit test for debug JSON output.

## Next step for real HomeKit permission testing

Configure signing in Xcode:

1. Open `HomeKitMCPHelper.xcodeproj`.
2. Select target `HomeKitMCPHelper`.
3. Signing & Capabilities → select a development team.
4. Ensure HomeKit capability is present.
5. Build and run normally.

Expected first-run behavior:

1. Launch the app.
2. macOS prompts for Apple Home access.
3. The app shows authorization status.
4. No Apple Home mutations are performed.
